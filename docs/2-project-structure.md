# 2. 프로젝트 구조 설계

## 여행 일정 공유 웹 서비스

Trip Planner Web

---

## 2.1 기능 구조

Trip Planner Web은 사용자가 여행 일정을 생성하고, 날짜별 세부 계획을 등록하며, 참여자와 함께 일정을 공유 및 수정할 수 있는 웹 서비스이다.

| 기능 영역 | 하위 기능 |
| --- | --- |
| 사용자 관리 | 회원가입, 로그인, 로그아웃, 세션 관리 |
| 여행 일정 관리 | 일정 생성, 목록 조회, 상세 조회, 수정, 삭제 |
| 날짜별 세부 일정 관리 | 세부 일정 등록, 날짜별 조회, 수정, 삭제, 예상 비용 관리 |
| 지도 및 위치 표시 | Kakao Map API 연동, 장소 위치 표시, 위도/경도 저장 |
| 공유 및 참여자 관리 | 공유 링크 생성, 참여자 초대, 권한 관리, 공동 편집 |

---

## 2.2 시스템 구조

본 프로젝트는 MVC(Model2) 패턴을 적용하여 화면 출력, 요청 처리, 데이터 처리를 분리한다.

| 계층 | 구성 요소 | 주요 역할 |
| --- | --- | --- |
| View | JSP, Bootstrap, JavaScript | 화면 출력, 입력 폼, 목록/상세 화면 구성, 지도 표시 |
| Controller | Servlet | 요청 URL 처리, 입력값 검증, 세션 확인, Model 호출, 화면 이동 제어 |
| Model | DTO, DAO | 데이터 객체 관리, JDBC 기반 CRUD 처리 |
| Database | MySQL | 사용자, 여행 일정, 세부 일정, 참여자 데이터 저장 |
| External API | Kakao Map API, Kakao Mobility API | 장소 위치 표시, 지도 기반 경로 확인 |

# 2. 프로젝트 구조 설계

## MVC 처리 흐름

| 순서 | 처리 단계 | 설명 |
| ---: | --- | --- |
| 1 | 사용자 요청 | 사용자가 JSP 화면에서 폼 제출 또는 버튼 클릭 |
| 2 | Servlet 처리 | Controller가 요청 URL, 입력값, 로그인 상태와 권한 확인 |
| 3 | DAO 호출 | DAO가 JDBC를 통해 MySQL에 데이터 저장 또는 조회 |
| 4 | 결과 반환 | Servlet이 처리 결과를 request/session에 저장 |
| 5 | 화면 출력 | JSP가 전달받은 데이터를 이용해 목록, 상세, 오류 메시지 출력 |

## 2.6 데이터베이스 구조

| 테이블 | 주요 필드 | 역할 |
| --- | --- | --- |
| `users` | `user_id`, `login_id`, `password`, `user_name`, `email`, `created_at` | 사용자 계정 정보 저장 |
| `trips` | `trip_id`, `user_id`, `trip_title`, `destination`, `start_date`, `end_date`, `share_code` | 여행 일정 기본 정보 저장 |
| `trip_details` | `detail_id`, `trip_id`, `schedule_date`, `place_name`, `visit_time`, `cost`, `latitude`, `longitude` | 날짜별 세부 일정과 위치 정보 저장 |
| `trip_members` | `member_id`, `trip_id`, `user_id`, `role`, `joined_at` | 참여자와 권한 정보 저장 |

---

## 테이블 관계

| 관계 | 설명 |
| --- | --- |
| `users` 1 : N `trips` | 한 사용자는 여러 여행 일정을 생성할 수 있다. |
| `trips` 1 : N `trip_details` | 하나의 여행 일정은 여러 날짜별 세부 일정을 가진다. |
| `users` 1 : N `trip_members` | 한 사용자는 여러 여행 일정에 참여할 수 있다. |
| `trips` 1 : N `trip_members` | 하나의 여행 일정은 여러 참여자를 가질 수 있다. |

외래키와 `ON DELETE CASCADE`를 사용하여 일정 삭제 시 관련 세부 일정과 참여자 정보가 함께 정리되도록 설계한다.

## 2.7 기능별 처리 흐름

### 회원가입 및 로그인

| 순서 | 처리 내용 |
| ---: | --- |
| 1 | 사용자가 회원가입 또는 로그인 화면에서 정보를 입력한다. |
| 2 | JSP가 입력 데이터를 `UserServlet`으로 전송한다. |
| 3 | Servlet은 필수값과 형식을 검증하고 `UserDAO`를 호출한다. |
| 4 | DAO는 `users` 테이블에 사용자 정보를 저장하거나 로그인 정보를 조회한다. |
| 5 | 성공 시 세션에 로그인 사용자 정보를 저장하고, 실패 시 오류 메시지를 출력한다. |

## 2.7 기능별 처리 흐름

### 여행 일정 생성 및 세부 일정 등록

| 순서 | 처리 내용 |
| ---: | --- |
| 1 | 사용자가 여행 정보 또는 세부 일정 정보를 입력한다. |
| 2 | JSP가 생성 또는 수정 요청을 Servlet으로 전송한다. |
| 3 | Servlet은 로그인 상태와 작성자·참여자 권한을 확인한다. |
| 4 | 권한이 있으면 DAO를 통해 `trips` 또는 `trip_details` 테이블에 저장한다. |
| 5 | 처리 후 여행 상세 화면으로 이동하여 변경된 일정을 표시한다. |

# 2.10 문서 정리 요약

- 프로젝트 구조는 사용자 관리, 여행 일정 관리, 날짜별 세부 일정 관리, 지도 표시, 공유 및 참여자 관리 기능을 중심으로 설계하였다.
- 시스템은 JSP / Servlet / DAO / DTO / MySQL로 구성된 MVC(Model2) 구조를 따른다.
- 역할 분담은 백엔드/DB 중심 담당과 프론트엔드/UI 중심 담당으로 나누되, 화면과 서버 연동이 필요한 기능은 공동으로 점검한다.
- 일정은 핵심 기능 구현, 지도 및 공유 기능 구현, 통합 테스트 순서로 계획하였다.
