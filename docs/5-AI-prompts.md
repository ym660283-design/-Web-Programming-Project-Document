## 5. F-01 회원가입 DB 연동 및 아이디 중복확인

날짜 : 2026년 06월 05일

모드 : Agent

지시사항 :

* `docs/2-project-structure.md`의 1.3 디렉터리 구조를 기준으로 회원 관련 요청 처리는 `controller/UserServlet.java`, DB 접근은 `dao/UserDAO.java`, 데이터 전달은 `dto/UserDTO.java`, DB 연결 및 초기화는 `util/DBUtil.java`에서 처리하도록 구성해줘.
* 회원가입 화면에서 제출된 아이디, 이름, 이메일, 비밀번호, 비밀번호 확인 값을 `POST /register`로 받아 서버에서 검증하고, 검증을 통과한 회원 정보를 MySQL `users` 테이블에 저장해줘.
* 아이디는 영문을 반드시 포함한 4~20자의 영문과 숫자만 허용하고, 숫자만 입력한 아이디는 서버와 화면에서 모두 사용할 수 없도록 처리해줘.
* XAMPP MySQL 환경에서 `travel_db` 데이터베이스가 아직 생성되지 않은 경우 `resources/schema.sql`을 이용해 DB와 테이블을 자동 생성한 뒤 회원가입 및 아이디 중복확인이 동작하도록 `DBUtil.java`를 보강해줘.
* 회원가입 화면의 아이디 입력칸 옆에 `중복확인` 버튼을 추가하고, `/user/check-login-id` 요청을 통해 실제 DB에 같은 아이디가 있는지 확인해 사용 가능 여부를 화면에 표시해줘.
* 아이디 중복확인을 완료한 뒤에만 회원가입을 제출할 수 있도록 하고, 중복확인 이후 아이디 값을 수정하면 확인 상태를 초기화해 다시 중복확인을 요구하도록 구현해줘.


## 6. F-02 로그인 및 로그아웃 기능 구현

날짜 : 2026년 06월 09일

모드 : Agent

지시사항 :

1. `docs/1-requirement-analysis.md`, `docs/2-project-structure.md`를 확인하고, 문서에 작성된 MVC(Model2) 구조와 회원 인증 요구사항에 맞게 로그인 및 로그아웃 기능을 구현해줘.
2. 회원 인증 관련 요청 처리는 `controller/UserServlet.java`, DB 접근은 `dao/UserDAO.java`, 사용자 데이터 전달은 `dto/UserDTO.java`, 세션 키 관리는 `util/SessionNames.java`에서 처리하도록 구성해줘.
3. 로그인 화면은 `views/user/login.jsp`에 구성하고, 공통 `header.jsp`, `footer.jsp`, `assets/css/style.css`를 사용하여 기존 TripLog 디자인과 Bootstrap 스타일을 유지해줘.
4. 로그인 폼에는 아이디(`login_id`)와 비밀번호(`password`) 입력 필드, 로그인 버튼, 회원가입 페이지 이동 링크를 배치하고, 필수 입력값이 비어 있으면 안내 메시지를 보여줘.
5. 로그인 요청은 `POST /login`으로 처리하고, `users` 테이블의 `login_id`, `password`와 일치하는 사용자가 있는지 `UserDAO`에서 조회해줘.
6. 로그인 성공 시 `SessionNames.LOGIN_USER`를 사용해 로그인 사용자 정보를 세션에 저장하고 여행 일정 목록 화면(`/trips`)으로 이동하도록 해줘.
7. 로그인 실패 시 `아이디 또는 비밀번호가 올바르지 않습니다.` 메시지를 로그인 화면에 출력하고, 어떤 값이 틀렸는지 구체적으로 노출하지 않도록 해줘.
8. 이미 로그인한 사용자가 `/login` 또는 `/register`에 접근하면 여행 일정 목록(`/trips`)으로 이동하도록 처리해줘.
9. 로그아웃 요청은 `POST /logout`으로 처리하고, 세션을 무효화한 뒤 `/login?logout=1`로 이동하여 로그아웃 완료 메시지를 보여줘.
10. 로그인하지 않은 사용자가 여행 일정 기능에 접근하면 `/login?required=1`로 이동시키고 로그인 필요 안내 메시지를 표시해줘.
11. JSP는 화면 출력 중심으로 유지하고, 로그인 검증과 세션 처리 같은 주요 로직은 Servlet과 DAO에서 처리되도록 정리해줘.

## 8. F-05 날짜별 세부 일정 화면 구현

날짜 : 2026년 06월 07일

모드 : Agent

지시사항 :

- 별도의 세부 일정 JSP를 생성하지 않고 기존 `views/trip/detail.jsp`에 여행 정보와 날짜별 세부 일정 조회 및 관리 화면을 함께 구성해줘.
- 여행 상세보기 요청은 `TripServlet`에서 분리하여 `TripDetailServlet`의 `/trip-details?trip_id={tripId}` 경로로 처리하고, 여행 기본 정보와 날짜별 예시 일정 데이터를 `detail.jsp`에 전달해줘.
- 여행 일정 목록의 상세보기 버튼도 `/trip-details` 요청으로 이동하도록 연결해줘.
- 여행 상세 화면의 세부 일정 조회 영역에는 여행 날짜별 선택 버튼과 선택한 날짜의 방문 시간, 장소, 메모, 예상 비용을 확인할 수 있는 타임라인을 표시해줘.
- 세부 일정 등록, 수정, 삭제 화면은 수정 권한이 있는 사용자를 위한 접이식 관리 영역으로 구성하고, 기본 상태에서는 닫혀 있도록 해줘.
- 관리 영역의 일정 날짜는 모든 날짜 버튼을 한 번에 표시하지 말고, 현재 선택 날짜를 표시하는 목록바를 누르면 여행 기간의 날짜 목록이 펼쳐지도록 구현해줘. 여행 기간이 긴 경우 목록 내부에서 스크롤할 수 있도록 구성해줘.
- 관리용 날짜 목록에서 날짜를 선택하면 해당 날짜의 등록 일정 관리 목록과 일정 개수가 변경되도록 구현해줘.
- 세부 일정 입력 폼에는 방문 장소, 방문 시간, 예상 비용, 메모 입력칸을 구성하고, 방문 장소와 방문 시간은 필수 입력 항목으로 처리해줘.
- 필수 입력값이 비어 있는 상태에서 등록 또는 수정 버튼을 누르면 입력칸의 테두리를 빨간색으로 표시하고 아래에 `필수 입력 칸입니다.`라는 작은 안내 문구를 출력해줘.
- 등록 일정 관리 목록의 수정 버튼을 누르면 해당 일정의 날짜, 장소, 시간, 비용, 메모를 입력 폼에 표시하고, `세부 일정 등록` 버튼을 `세부 일정 수정` 버튼으로 변경해줘.
- 수정 모드에서는 `수정 취소` 버튼을 표시하고, 클릭하면 입력값과 수정 대상 정보를 초기화한 뒤 등록 모드로 돌아가도록 구현해줘.
- `입력 초기화` 버튼은 현재 등록 또는 수정 모드를 유지하면서 방문 장소, 방문 시간, 예상 비용, 메모 입력값만 초기화하도록 구현해줘.
- 브라우저에서 처리해야 하는 날짜 선택, 입력 검증, 수정 모드 전환, 수정 취소 및 입력 초기화 동작은 `assets/js/trip.js`에서 처리해줘.


## 9. F-04 여행 일정 CRUD 기능 구현

날짜 : 2026년 06월 09일

모드 : Agent

지시사항 :

1. `docs/1-requirement-analysis.md`, `docs/2-project-structure.md`를 확인하고, 문서에 작성된 MVC(Model2) 구조와 여행 일정 관리 요구사항에 맞게 구현해줘.
2. 여행 일정 기능은 JSP / Servlet / DAO / DTO 구조로 분리해서 구현해줘.
   데이터 전달 객체는 `dto/TripDTO.java`, DB 접근 로직은 `dao/TripDAO.java`, 요청 처리는 `controller/TripServlet.java`에서 담당하도록 해줘.
3. `trips` 테이블 구조에 맞춰 여행 일정 생성, 조회, 수정, 삭제 기능을 구현해줘.
   필드는 `trip_id`, `user_id`, `trip_title`, `destination`, `start_date`, `end_date`, `description`, `share_code`를 기준으로 처리해줘.
4. 로그인한 사용자만 여행 일정 기능을 사용할 수 있도록 하고, 로그인하지 않은 사용자가 접근하면 로그인 화면으로 이동하도록 처리해줘.
5. 여행 일정 목록 화면에서는 로그인 사용자가 작성했거나 참여 중인 여행 일정만 DB에서 조회해 출력하도록 수정해줘.
6. 여행 일정 생성 화면에서는 제목, 목적지, 시작일, 종료일, 설명을 입력받아 `POST /trips` 요청으로 저장되도록 연결해줘.
   저장 성공 후에는 해당 여행 일정 상세 화면으로 이동하도록 해줘.
7. 여행 일정 수정은 작성자만 가능하도록 해줘.
   수정 화면은 기존 `views/trip/form.jsp`를 공용으로 사용하고, 기존 일정 정보를 입력폼에 표시한 뒤 수정 내용을 DB에 반영해줘.
8. 여행 일정 삭제도 작성자만 가능하도록 해줘.
   삭제 요청은 `POST /trips`에서 처리하고, 삭제 후 여행 일정 목록 화면으로 이동하도록 해줘.
9. 여행 일정 상세 화면은 `/trip-details?trip_id=...` 요청으로 조회하고, DB에서 여행 기본 정보를 불러와 `views/trip/detail.jsp`에 표시되도록 해줘.
   여행 기간에 맞춰 날짜 목록도 실제 시작일과 종료일 기준으로 표시되도록 해줘.
10. 기존 화면 디자인과 공통 레이아웃은 유지하되, JSP는 화면 출력 중심으로 두고 주요 입력 처리와 권한 검사는 Servlet에서 처리되도록 정리해줘.

5. 공유 링크로 접속한 로그인 사용자에게 여행 제목, 작성자, 목적지, 여행 기간을 보여주고 일정에 참여할 것인지 확인해줘.
6. 사용자가 참여에 동의한 경우에만 `trip_members` 테이블에 `viewer` 권한으로 추가하고, 동의하지 않은 사용자는 참여자로 등록하지 않도록 해줘.
8. 참여자 권한은 `viewer`와 `editor`로 구분하고, 여행 일정 작성자가 참여자 관리 화면에서 권한을 변경하거나 참여자를 내보낼 수 있도록 해줘.
9. `viewer`는 여행 일정과 세부 일정을 조회할 수만 있도록 하고, `editor`는 여행 기본 정보와 날짜별 세부 일정을 생성, 수정, 삭제할 수 있도록 Servlet과 DAO에서 권한을 검사해줘.
10. 여행 일정 삭제, 공유 링크 관리, 참여자 권한 변경과 내보내기는 작성자만 가능하도록 제한해줘.
11. 권한에 따라 여행 목록과 상세 화면의 수정, 삭제, 공유 및 참여자 관리 버튼을 구분하여 표시하고, 참여자에게 현재 자신의 권한을 보여줘.
12. 브라우저 뒤로가기로 작성자 전용 버튼이 남아 있는 경우에도 서버에서 권한을 다시 검사하고, 권한이 없으면 오류 페이지 대신 여행 상세 화면으로 이동하여 작성자만 이용할 수 있다는 경고 메시지를 표시해줘.
13. 공유 링크를 재발급하면 기존 링크가 더 이상 동작하지 않고 새 링크만 사용할 수 있도록 `trips.share_code` 값을 변경해줘.
14. `share_code`의 DB `UNIQUE` 제약을 유지하고, 일정 생성이나 공유 링크 재발급 중 MySQL 중복 키 오류가 발생하면 새로운 공유 코드를 생성하여 최대 5회까지 재시도해줘.
=======
1. `docs/1-requirement-analysis.md`, `docs/2-project-structure.md`, `docs/3-feature-Implementation.md`를 확인하고, 문서에 작성된 Kakao Map API 연동 요구사항에 맞게 여행 상세 화면의 지도 기능을 구현해줘.
2. 여행 상세 화면에서 세부 일정의 위치 정보를 지도에 표시할 수 있도록 `views/trip/detail.jsp`에 지도 영역을 추가하고, Kakao JavaScript SDK 스크립트를 `web.xml`의 JavaScript 키 설정값을 사용해 로드하도록 구성해줘.
3. 세부 일정 등록 및 수정 폼에서 방문 장소명을 기준으로 Kakao 장소 검색을 수행하고, 사용자가 검색 결과에서 실제 장소를 선택하면 해당 장소의 위치 정보가 함께 저장되도록 구현해줘.
4. `trip_details` 테이블에 위치 정보를 저장할 수 있도록 위도와 경도 컬럼을 추가하고, 기존 DB에도 컬럼이 없으면 자동 보강되도록 DB 초기화 로직을 수정해줘.
5. 위치 정보 처리는 `TripDetailDTO`, `TripDetailDAO`, `TripDetailServlet`에 반영하고, JSP에는 위도와 경도 입력값이 직접 노출되지 않도록 hidden 필드로 관리해줘.
6. 여행 상세 화면의 선택된 날짜에 위치가 저장된 세부 일정이 있으면 장소별 마커를 표시하고, 마커 클릭 시 방문 시간, 장소명, 예상 비용, 메모를 확인할 수 있도록 해줘.
7. 등록된 장소가 여러 개일 경우 방문 순서에 따라 마커 번호를 표시하고, 장소 간 이동 동선을 지도 위에 표시해줘.
8. 실제 도로 기준 경로와 소요시간을 제공하기 위해 Kakao Mobility 길찾기 REST API를 서버에서 호출하는 `/trip-route` Servlet을 추가해줘. REST API 키는 브라우저 JavaScript에 노출하지 않고 서버 설정에서만 사용하도록 구성해줘.
9. `/trip-route`는 로그인한 사용자가 접근 가능한 여행 일정인지 확인한 뒤, 선택 날짜의 위치 저장 세부 일정을 방문 시간 순서로 조회하고, 인접한 장소 간 실제 도로 경로 JSON을 반환하도록 구현해줘.
10. `assets/js/trip.js`에서는 `/trip-route` 응답의 도로 좌표를 Kakao Map Polyline으로 표시하고, 총 이동거리와 예상 소요시간, 구간별 이동거리와 소요시간을 화면에 출력해줘.
11. Kakao Mobility API 호출이 실패하거나 실제 도로 경로를 가져오지 못하는 경우에는 기존 위치 좌표를 이용한 직선 동선과 직선거리 기준 예상시간으로 fallback 되도록 처리해줘.
12. 구간별 방문 시간 간격을 계산하여 다음 일정까지의 간격이 실제 이동시간보다 짧으면 해당 구간에 `일정의 간격이 이동시간에 비해 짧습니다` 안내 문구가 표시되도록 구현해줘.
13. 지도, 마커 번호, 경로 요약, 이동시간 안내, 일정 간격 경고 문구는 기존 TripLog 디자인과 어울리도록 `assets/css/style.css`에 스타일을 추가해줘.
14. 구현 후 Maven 빌드가 성공하는지 확인하고, 로컬 Jetty 서버에서 여행 상세 화면, 정적 리소스, `/trip-route` API가 정상 응답하는지 검증해줘.
